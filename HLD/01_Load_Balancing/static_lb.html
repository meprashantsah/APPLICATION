Static Load Balancing

Algorithms that distribute traffic without considering the current state of the servers. Simple, fast, and predictable — but not adaptive.

1. Round Robin

Requests are distributed sequentially across the server pool. Server 1 gets request 1, Server 2 gets request 2, and so on. After the last server, it loops back to the first.

🔑 Key Idea Think of it like dealing cards to players — each player gets one card at a time in rotation.

Example Scenario

Imagine 3 servers: S1, S2, S3. Here is how 6 requests are distributed:

Request # Server Assigned Logic
1S1Start from first server
2S2Move to next
3S3Move to next
4S1Loop back to first
5S2Continue rotation
6S3Continue rotation

Simple Code Simulation

import java.util.List;

public class RoundRobinLoadBalancer {
    private final List servers;
    private int currentIndex = 0;

    public RoundRobinLoadBalancer(List servers) {
        this.servers = servers;
    }

    public synchronized String getServer() {
        String server = servers.get(currentIndex);
        currentIndex = (currentIndex + 1) % servers.size();
        return server;
    }

    public static void main(String[] args) {
        List servers = List.of("S1", "S2", "S3");
        RoundRobinLoadBalancer lb = new RoundRobinLoadBalancer(servers);

        for (int i = 1; i <= 6; i++) {
            System.out.println("Request " + i + " → " + lb.getServer());
        }
    }
}

// Output:
// Request 1 → S1
// Request 2 → S2
// Request 3 → S3
// Request 4 → S1
// Request 5 → S2
// Request 6 → S3
⚠️ When Round Robin Fails If S1 is a powerful machine and S3 is a weak one, Round Robin treats them equally. A heavy request going to S3 might crash it while S1 sits idle.

2. Weighted Round Robin

Assigns a "weight" to each server based on its capacity. A server with weight 3 gets 3x more requests than a server with weight 1.

Server Weight Requests per cycle
S1 (Powerful)33 requests
S2 (Medium)22 requests
S3 (Weak)11 request

A full cycle of 6 requests would look like: S1, S1, S1, S2, S2, S3

3. IP Hash

The client's IP address is hashed to determine which server receives the request. The same client always hits the same server. 

import java.util.List;

public class IpHashLoadBalancer {
    private final List servers;

    public IpHashLoadBalancer(List servers) {
        this.servers = servers;
    }

    public String getServer(String clientIp) {
        int hashValue = clientIp.hashCode();
        int index = Math.abs(hashValue) % servers.size();
        return servers.get(index);
    }

    public static void main(String[] args) {
        List servers = List.of("S1", "S2", "S3");
        IpHashLoadBalancer lb = new IpHashLoadBalancer(servers);

        System.out.println(lb.getServer("192.168.1.10"));  // → S1
        System.out.println(lb.getServer("192.168.1.20"));  // → S1
        System.out.println(lb.getServer("192.168.1.30"));  // → S1

        // Note: All 3 clients might hash to the same server!
        // That's why consistent hashing is preferred in practice.
    }
}
✅ Pros of Static LB
  • Very simple to implement and understand
  • No overhead of tracking server health or load
  • Predictable behavior — easy to debug
  • IP Hash provides natural session affinity
❌ Cons of Static LB
  • Ignores server health — sends traffic to crashed servers
  • Ignores current load — a busy server gets the same traffic as an idle one
  • IP Hash can create hotspots (many clients hashing to same server)
  • Not suitable for servers with different capacities (without weights)

Quick Comparison

Algorithm Best For Avoid When
Round Robin Homogeneous servers, equal load Servers have different capacities
Weighted Round Robin Heterogeneous servers Server load fluctuates rapidly
IP Hash Need session persistence Clients are behind NAT (same IP)
📝 Interview Tip In interviews, always mention: "Round Robin is a good starting point, but in production we usually use dynamic algorithms like Least Connections because they adapt to real-time conditions."
