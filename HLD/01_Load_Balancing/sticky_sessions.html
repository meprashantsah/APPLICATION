Sticky Sessions (Session Affinity)

Ensures all requests from a specific client are routed to the same server throughout their session. Critical for applications that store session data locally on the server.

Why Do We Need Sticky Sessions?

Imagine a shopping cart application:

  1. User adds Item A to cart → Request goes to Server 1 → Cart stored in Server 1's memory
  2. User adds Item B to cart → Request goes to Server 2 → Server 2 has no cart data!
  3. User sees only Item B in cart → Data is lost!
🚨 The Problem Without sticky sessions, each request from the same user might hit a different server. If session data is stored locally (in-memory), the user experience breaks.

How Sticky Sessions Work

The load balancer uses a "sticky key" to identify the client and always route them to the same server.

Common Sticky Keys

MethodWhat It UsesProsCons
IP Hash Client IP address Simple, no cookies needed Breaks with NAT (many users share one IP)
Cookie-Based Browser cookie (e.g., SERVERID=S1) Works through NAT/proxies Requires cookie support, privacy concerns
URL Parameter Query string (e.g., ?server=S1) Works without cookies Ugly URLs, SEO issues

Cookie-Based Sticky Sessions (Most Common)

Here's how it works step by step:

# Step 1: First Request (no cookie yet)
Client ──► Load Balancer ──► Server 1
                              └── Creates session
                              └── Sets cookie: SERVERID=S1

# Step 2: Load Balancer adds cookie to response
Client ◄── Load Balancer ◄── Server 1
     (receives SERVERID=S1 cookie)

# Step 3: Second Request (client sends cookie)
Client ──► Load Balancer ──► Server 1
     (sends SERVERID=S1)        └── Reads cookie
                              └── Routes to S1

# Step 4: All future requests go to S1
Client ──► Load Balancer ──► Server 1
     (SERVERID=S1)              └── Same session!

Example: Nginx Configuration

upstream backend {
    server 10.0.0.1:8080;
    server 10.0.0.2:8080;
    server 10.0.0.3:8080;
}

server {
    location / {
        proxy_pass http://backend;
        
        # Enable sticky sessions with cookie
        sticky cookie srv_id expires=1h domain=.example.com path=/;
    }
}

What Happens When the Sticky Server Goes Down?

This is the biggest challenge with sticky sessions.

ScenarioWhat HappensSolution
Server crashes User's session is lost Session replication across servers
Server removed for maintenance User gets redirected to new server Graceful session migration
Server overloaded User stuck on slow server Break stickiness for health reasons
💡 Better Alternative: Session Externalization Instead of storing sessions on individual servers, store them in a shared store like Redis or Memcached. Then sticky sessions become unnecessary — any server can handle any request because session data is centralized.

Code Example: Simulating Sticky Sessions

import java.util.*;

public class StickyLoadBalancer {
    private final List servers;
    private final Map sessionMap = new HashMap<>();

    public StickyLoadBalancer(List servers) {
        this.servers = servers;
    }

    public synchronized Server route(String userId) {
        Server server;
        if (sessionMap.containsKey(userId)) {
            // Existing user → same server
            server = sessionMap.get(userId);
            System.out.println("User " + userId + " → " + server.name + " (sticky)");
        } else {
            // New user → pick least loaded server
            server = Collections.min(servers, Comparator.comparingInt(s -> s.load));
            sessionMap.put(userId, server);
            server.load++;
            System.out.println("User " + userId + " → " + server.name + " (new)");
        }
        return server;
    }

    public static void main(String[] args) {
        List servers = Arrays.asList(
            new Server("S1", 0),
            new Server("S2", 0),
            new Server("S3", 0)
        );

        StickyLoadBalancer lb = new StickyLoadBalancer(servers);
        lb.route("user_101");  // New → S1
        lb.route("user_102");  // New → S2 (S1 has 1 connection)
        lb.route("user_101");  // Existing → S1 (sticky!)
    }

    static class Server {
        String name;
        int load;

        Server(String name, int load) {
            this.name = name;
            this.load = load;
        }
    }
}

When to Use (and When NOT to Use)

✅ Use Sticky Sessions When:
  • Session data is stored in server memory (legacy apps)
  • Real-time connections (WebSockets) need the same server
  • File uploads are processed in chunks across multiple requests
  • You can't change the application architecture right now
❌ Avoid Sticky Sessions When:
  • You can use external session stores (Redis, DB)
  • You need true horizontal scaling (sticky = less scalable)
  • Server failures would cause significant user disruption
  • You want simple, stateless server architecture
📝 Interview Tip "I prefer external session stores over sticky sessions because they make the system truly stateless and horizontally scalable. Sticky sessions are a workaround for legacy systems that can't be refactored easily."
